# Prompt 12: VIBE Tab — Wire Terminal Output to Bridge

## CONTEXT

Prompt 11 built the VIBE tab. It renders, the `VibeTerminalBridge` creates a `TerminalTabItem` and enqueues `claude` + `/plan` commands, but the terminal output never reaches the bridge. The output stream shows "Starting Claude in plan mode..." forever.

**Root cause:** `VibeTerminalBridge.handleOutput()` exists but nothing calls it. The bridge creates a `TerminalTabItem` (the planning tab) but that tab item is never attached to a `LocalProcessTerminalView` with a shell process, because the VIBE tab view doesn't render a `TerminalSessionView` for it.

In the regular IDE, `TerminalTabView` → `TerminalSessionView` (NSViewRepresentable) creates the `MonitoringTerminalView`, starts the shell process, and hooks `onOutput`. The VIBE tab skips all of this — it only has a `Text()` view for display.

**The fix:** The VIBE tab needs to host an invisible `TerminalSessionView` (or equivalent) that actually runs the shell process. The terminal itself doesn't need to be visible — the VIBE tab's styled `Text` output view replaces it — but the process and output pipeline must exist.

## WHAT EXISTS (do NOT break)

- `VibeTabView` — the main VIBE UI with output stream, input, Lock Plan, Execute
- `VibeTerminalBridge` — creates `TerminalTabItem`, has `handleOutput()` and `handleExecutionOutput()`
- `TerminalSessionView` (NSViewRepresentable) — creates `MonitoringTerminalView`, starts shell, hooks `onOutput`
- `MonitoringTerminalView` — subclass of `LocalProcessTerminalView`, overrides `dataReceived()` to call `onOutput`
- `TerminalTabItem.attach()` — attaches a `LocalProcessTerminalView`, sends pending commands
- `VibeSummarizerService.handleGhostOutput()` — needs output routed to it for ghost tabs
